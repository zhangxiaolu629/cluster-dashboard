2. 客户端组件只保留“必须交互”的部分（表格筛选、按钮、主题切换）

## 你这个项目的具体改造建议（按优先级）

### 1) 先做低风险改造：页面文件全部服务端化

适用文件：`src/app/page.tsx`、`src/app/cluster/[id]/**/page.tsx`

**现状特征**：这些 `page.tsx` 主要是“取 params + 包壳 + 渲染组件”，几乎没浏览器 API。  
**做法**：

- 删除 `'use client'`
- 删除 `use(params)`，改成 `async` page + `await params`
- 继续把 `id` 作为普通 props 传给客户端组件

这一步改动小、风险低，能立刻把“路由入口”从 client bundle 中拿出来。

---

### 2) 把 `PageLayout` 拆成“轻客户端壳 + 页面内容”

`PageLayout` 现在是一个大客户端组件，还内置了 `renderContent` 分支（`NamespaceList/EventList/...`）。  
建议拆成：

- `AppShellClient`（客户端）：只负责
  - 侧边栏折叠
  - 主题切换
  - 菜单点击跳转
- 页面内容放回各自 `page.tsx`（服务端决定渲染哪个模块）

**收益**：避免把所有列表模块都“潜在打包进壳层依赖图”，边界更清晰，后续更容易做按路由分割。

---

### 3) 列表页数据改为“服务端取数 + 客户端展示”

例如 `NamespaceList`、`DeploymentList` 现在在浏览器里 `fetch("/api/...")`。  
建议：

- 新建服务端数据层（例如 `src/server/k8s/queries.ts`）
- `page.tsx`（Server）里拿数据后传给 `NamespaceListClient`（Client）
- `NamespaceListClient` 专注交互（搜索、分页、列排序）

> 这一步是体感性能提升最大的（首屏更快、加载闪烁更少、SEO/可观测也更好）。

---

### 4) API 路由保留给“写操作”和外部调用

你现在很多 `/api/*` 是给页面自己读数据用。  
改造后建议：

- 页面 SSR 用 `src/server/*` 直接调 `k8sFetch`（不绕 `/api`）
- `/api/*` 主要保留给：
  - 表单提交（写操作）
  - 需要前端轮询/动态刷新的接口
  - 外部系统调用入口

---

### 5) 加入“集群上下文强约束”

你当前路由是 `/cluster/[id]/...`，但部分 `/api/*` 没显式带 `clusterId`。  
建议在服务端数据层统一要求 `clusterId`，避免逻辑串群（这是业务正确性问题，优先级高于纯性能）。

---

## 推荐实施顺序（两次迭代）

### 迭代 A（1 天，低风险）

- 全部 `page.tsx` 改为 Server Component
- 去掉 `use(params)` 模式
- 保持现有 `PageLayout` 不拆，仅保证页面入口服务端化

### 迭代 B（2~3 天，中等风险，高收益）

- 拆分 `PageLayout`
- 列表页改成“Server 取数 + Client 展示”
- 抽 `src/server/k8s` 服务层

- Server Component 不能用 `useRouter/useEffect/window/localStorage`
- 从 Server 传到 Client 的 props 必须可序列化
- 如果还需要客户端实时刷新，可局部保留 client fetch，不必“一刀切”

_**Agent (default)**_

不会，**浏览器不会直接请求 K8s 集群**。

你现在改造后的链路是：

- **首屏渲染数据**：`page.tsx` 在服务端调用 `k8sFetch` / `@volcengine/openapi` 拉数据，再把结果作为 props 下发给客户端组件。
- **交互请求（筛选、刷新等）**：浏览器请求的是你自己的 Next 接口（`/api/*`），例如 `/api/deployments`、`/api/events`。
- **真正访问 K8s**：发生在服务端（`src/lib/k8s.ts` 的 `k8sFetch`），由服务端带 `K8S_TOKEN` 去请求 `K8S_API_SERVER`。

所以 Token 和 K8s 地址都在服务端环境变量里，不会暴露给浏览器（除非你在前端代码里显式打印/透传，这里没有这么做）。
